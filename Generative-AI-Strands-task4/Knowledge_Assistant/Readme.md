# Knowledge Assistant — Agentic RAG (Task 4, Part 1)

A Strands Agent that answers questions grounded in a set of PDFs, citing
its sources (file + page), and admitting when it can't find an answer.
Retrieval is exposed as a **tool** the agent decides to call — and can
call again with a refined query — rather than always running before the
model sees the question.

## Folder structure

```
KnowledgeAssistant/
│
├── app/
│   ├── __init__.py
│   ├── config.py        # paths, model IDs, chunking params, system prompt
│   ├── utils.py          # PDF/.md/.txt readers, chunking, section detection
│   ├── embeddings.py      # Titan Text Embeddings V2 client (Bedrock)
│   ├── ingestion.py        # orchestrates load -> chunk -> embed -> index
│   ├── vector_store.py      # FAISS index build/save/load/search
│   ├── retrieval.py          # query embedding + search + result formatting
│   ├── tools.py                # @tool retrieve_knowledge (wraps retrieval.py)
│   └── agent.py                 # builds the Strands Agent (Bedrock Claude)
│
├── docs/                # your document set (PDF, .md, .txt all supported)
│   ├── sample.pdf         # embeddings & vector search
│   └── guide.pdf           # chunking, retrieval, agentic RAG
│
├── vector_db/            # FAISS index + metadata, generated by ingest.py
│
├── ingest.py            # CLI: build/rebuild the index
├── main.py              # FastAPI app: serves the assistant over HTTP
├── requirements.txt
├── .env.example
└── README.md
```

## Topic / document set

`docs/sample.pdf` and `docs/guide.pdf` are short notes I wrote covering
the concepts this task is built on: embeddings, cosine similarity,
chunking strategy, metadata, and what makes RAG "agentic." Drop in your
own PDFs, `.md`, or `.txt` files to point the assistant at any other
topic — `ingest.py` picks up everything in `docs/`.

## Architecture

```
docs/*.pdf, *.md, *.txt
        │  app.utils: read + chunk (per-page for PDFs)
        ▼
app.ingestion ──► app.embeddings (Titan V2, Bedrock) ──► app.vector_store (FAISS, vector_db/)
                                                                  │
question ──► app.agent (Strands Agent, Claude on Bedrock)         │
                   │  calls tool                                   │
                   ▼                                                │
        app.tools.retrieve_knowledge ──► app.retrieval ─────────────┘
                   │
                   ▼
        grounded answer + citations (file + page)
```

- **Embedding model:** Amazon Titan Text Embeddings V2
  (`amazon.titan-embed-text-v2:0`) via Bedrock — no local model download,
  runs in the same AWS account/region as the agent's LLM calls, and
  supports configurable output dimensions (1024 by default).
- **Vector store:** FAISS (`IndexFlatIP`), local, single file under
  `vector_db/`. Titan embeddings are returned normalized, so inner
  product == cosine similarity.
- **Agent model:** Claude on Amazon Bedrock via `strands.models.BedrockModel`.
- **PDF handling:** `app/utils.py` reads PDFs page-by-page with `pypdf`
  and chunks each page independently, so every chunk keeps an accurate
  page number for citations.

## Why agentic, not plain RAG

The system prompt (`app/config.py`) instructs the model to:
1. Always retrieve before answering a factual question (never answer from
   general knowledge alone).
2. Judge whether the retrieved chunks are sufficient; if not, retrieve
   again with a refined query (up to `MAX_RETRIEVAL_ATTEMPTS`, default 3).
3. Say plainly that the knowledge base doesn't have the answer if nothing
   relevant turns up after retrying — instead of guessing.
4. Cite the source file and page for every claim.

## Setup

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Edit `.env` if you want to change region, model IDs, or chunking
parameters (defaults work out of the box). Configure AWS credentials
with Bedrock access (env vars, `aws configure`, or an IAM role), and make
sure model access is enabled in your Bedrock console for both Titan Text
Embeddings V2 and the Claude model you set as `AGENT_MODEL_ID`.

## Build the index

```bash
python ingest.py
```

Re-run this any time you add/change files in `docs/` — it rebuilds
`vector_db/index.faiss` and `vector_db/chunks.json` from scratch.

## Ask questions

Start the API:
```bash

python -m uvicorn main:app --reload
```

Interactive docs (Swagger UI) at `http://127.0.0.1:8000/docs`.

Ask a question:
```bash
curl -X POST http://127.0.0.1:8000/ask \
     -H "Content-Type: application/json" \
     -d '{"question": "What is agentic RAG?"}'
```

Response:
```json
{
  "question": "What is agentic RAG?",
  "answer": "In plain RAG, retrieval always runs before the model sees the question... (source: guide.pdf, page 2)"
}
```

Health check: `GET /health` → `{"status": "ok"}`.

If you call `/ask` before running `ingest.py`, the API returns a `503`
with a message telling you to build the index first.

## Example questions and answers

*(Representative of what the agent returns; exact wording will vary run to run.)*

**Q1: What is agentic RAG and how is it different from plain RAG?**
> In plain RAG, retrieval always runs before the model sees the question.
> In agentic RAG, retrieval is exposed as a tool, and the model decides
> whether to retrieve, what to search for, and whether one pass is
> enough (source: guide.pdf, page 2).

**Q2: Why use cosine similarity instead of Euclidean distance for text embeddings?**
> Cosine similarity measures the angle between vectors rather than their
> magnitude, so it isn't thrown off by differences in text length — this
> is why it's preferred for text embedding models (source: sample.pdf, page 1).

**Q3: What chunk size and overlap should I start with?**
> A reasonable starting point is 500-1000 characters per chunk with a
> 10-20% overlap between consecutive chunks, so information near a chunk
> boundary isn't lost (source: guide.pdf, page 1).

**Q4: What metadata should I store with each chunk, and why?**
> Each chunk should carry its source file name, section/heading, and
> position in the document (e.g. page number). This is what lets the
> assistant cite exactly where an answer came from (source: guide.pdf, page 1).

**Q5: Does Strands support OpenAI's GPT models, or only Bedrock?**
> The knowledge base doesn't contain information about which model
> providers Strands supports — I couldn't find an answer to this in the
> documents. *(demonstrates the "not found" behavior — this question is
> outside the ingested document set on purpose)*
